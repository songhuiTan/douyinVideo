# Design: TikTok 视频拆解网站 — 爆款复刻指南生成器

Generated by /office-hours on 2026-04-06
Branch: main
Repo: douyinVideo
Status: APPROVED
Mode: Builder (Hackathon)

## Problem Statement

TikTok/抖音创作者每天看到爆款视频，想复刻但不知道"怎么拍的"。现有工具只能做数据分析（播放量、互动率），没有人告诉你"这个视频的分镜是什么、用了什么拍摄技巧、每秒的节奏怎么设计"。

## What Makes This Cool

上传一个爆款视频，30 秒后拿到一份**完整的复刻指南**：分镜表、每段拍摄技巧、配乐建议、节奏标注、可直接使用的提示词。用户拿走就能拍。

核心"哇"时刻：创作者第一次看到一个 AI 把视频"解剖"成可执行的拍摄蓝图。

## Key Definitions

- **提示词逆向（Prompt Reversal）**: 给定视频的分析结果，反推出创作者可能使用的创意简报或 AI 提示词。例如：从画面和字幕推断出"这段用了悬念开头 + 快速转场 + 情绪反转"的创意策略。
- **复刻指南（Replication Guide）**: 一份可执行的拍摄蓝图，包含：分镜表、每段拍摄技巧、配乐建议、节奏标注、可直接使用的提示词。用户拿走就能拍。

## Constraints

- 黑客松/Demo 场景，时间有限
- 技术栈：Next.js + Python FastAPI
- AI 服务：Gemini 2.5 Flash（视频理解）+ gpt-4o-mini-transcribe（字幕转写）+ Claude Sonnet 4（文本推理）
- MVP 不做视频生成，只做分析+输出复刻指南
- 必须包含分段拆解功能（用户明确要求）
- 视频格式：MP4/H.264，9:16 竖屏优先（兼容 16:9），最大 500MB，15-120 秒
- 单用户场景（黑客松 Demo，不做并发控制，后续加 Celery 队列）

## Premises

1. ✅ 创作者有素材但缺方法 — "为什么别人的能火"是真实痛点
2. ✅ "复刻指南" > "分析报告" — 用户要的是"怎么做"不是"为什么"
3. ✅ Gemini 2.5 Flash 原生视频理解能力足够完成场景识别和内容理解
4. ❌ 不跳过分段 — 分段拆解是核心展示能力，必须保留

## Approaches Considered

### Approach A: Gemini + PySceneDetect 分段（CHOSEN）
- PySceneDetect 本地场景检测 → 按场景分段
- 每段上传 Gemini 2.5 Flash 原生视频理解
- gpt-4o-mini-transcribe 做字幕转写
- Claude Sonnet 4 做提示词逆向 + 爆款模式提炼
- Effort: M | Risk: Low | Completeness: 8/10

### Approach B: 纯 Gemini 端到端
- 整个视频一次上传 Gemini，一个 prompt 出结果
- Effort: S | Risk: Med | Completeness: 5/10
- Rejected: 对 prompt 要求太高，长视频可能超 token 限制

### Approach C: 双模型竞争
- Gemini 视频理解 + Claude 分析字幕，交叉验证
- Effort: L | Risk: Med | Completeness: 9/10
- Rejected: 黑客松时间不够

## Recommended Approach

**Approach A: Gemini + PySceneDetect 分段**

核心管线：
```
上传视频 → PySceneDetect 分段 → 每段 Gemini 视频理解
         → ffmpeg 提取音频 → gpt-4o-mini-transcribe 字幕
         → Claude 推理：提示词逆向 + 爆款模式提炼
         → 生成完整复刻指南
```

AI 服务分工（基于 4-6 段/60s 视频的假设，快剪辑视频可能达 15-20 段）：
| 任务 | 模型 | 成本/60s视频（4-6段） |
|------|------|:----------:|
| 视频理解（画面+内容+节奏） | Gemini 2.5 Flash | ~$0.01-0.05 |
| 音频转写（字幕/台词） | gpt-4o-mini-transcribe | ~$0.003 |
| 提示词逆向 + 模式提炼 | Claude Sonnet 4 | ~$0.05-0.10 |
| **合计** | | **~$0.06-0.15** |

> 快剪辑视频（15-20 段）成本可能达到 $0.30-0.50。建议对超过 10 段的视频合并相邻短段。

## Error & Degradation

| 场景 | 降级策略 |
|------|---------|
| PySceneDetect 检测到 0 段 | 整片作为 1 段处理 |
| 视频无音轨 | 跳过转写步骤，纯视觉分析 |
| Gemini API 超时 | 重试 1 次，仍失败则标记该段为"分析失败" |
| 转写 API 失败 | 用 Gemini 的音频理解能力替代 |
| 输出格式错误 | 重试 1 次，仍失败则降级为纯文本输出 |

## Success Criteria

1. 用户能上传一个 TikTok/抖音短视频（15-120 秒）
2. 系统自动将视频切割为场景片段（PySceneDetect）
3. 每个片段展示：字幕 + 画面分析 + 拍摄技巧
4. 生成完整的"复刻指南"：分镜表 + 提示词 + 节奏标注 + 配乐建议
5. 指南可以一键复制或导出
6. 端到端处理时间 < 60 秒（60 秒视频）

### Section 1: Architecture Review

**Findings: 6 issues (2 P1, 3 P2, 1 P3)**

**[P1] (confidence: 9/10) Missing Pipeline Orchestrator**
Design specifies a multi-step pipeline (segment → Gemini → transcribe → Claude) but no orchestrator service exists. The core value of this product lives in the pipeline coordination logic. Without it, the 3 existing services (video_processor, gemini_analyzer, schemas) are disconnected islands.

Decision: Create `backend/services/pipeline.py` as the orchestrator. It coordinates:
1. Scene detection → segment list
2. Parallel: Gemini analysis (asyncio.gather) + audio extraction + transcription
3. Claude reasoning (prompt reversal + pattern extraction)
4. Assembly into ReplicationGuide

**[P1] (confidence: 9/10) Missing Transcription + Claude Services**
Design specifies gpt-4o-mini-transcribe and Claude Sonnet 4, but neither service file exists. `requirements.txt` lists the packages but no code uses them.

Decision: Create `backend/services/transcriber.py` (OpenAI Whisper) and `backend/services/claude_reasoner.py` (Anthropic). These are the two remaining AI service clients.

**[P2] (confidence: 8/10) Synchronous Gemini Code Blocks Event Loop**
`gemini_analyzer.py:26-29` uses `time.sleep(1)` polling loop. In a FastAPI async context, this blocks the entire server. For N parallel segment analyses, all parallel calls would serialize.

Decision: Wrap in `asyncio.to_thread()` or use `httpx` async client directly with Gemini REST API. For hackathon MVP, `asyncio.to_thread()` is the fastest fix.

**[P2] (confidence: 7/10) No FastAPI App Entry Point**
`backend/api/` directory exists but is empty. No `main.py`, no routes, no CORS config. The Next.js frontend needs CORS to call FastAPI.

Decision: Create `backend/main.py` with FastAPI app, CORS middleware, and 4 routes: upload, analyze, status, result.

**[P2] (confidence: 7/10) No State Management Between Requests**
Design mentions SQLite but no DB code exists. For hackathon single-user demo, in-memory dict keyed by video_id is sufficient. Add SQLite later for persistence.

Decision: Use in-memory dict `analysis_store: dict[str, AnalysisState]` for MVP. No SQLite for hackathon.

**[P3] (confidence: 6/10) Segment Merging Threshold Undefined**
Design says "超过 10 段合并相邻短段" but doesn't define "short". Without a threshold, a 20-segment video stays at 20 Gemini calls ($0.50+ cost).

Decision: Merge segments shorter than 2 seconds with their neighbor. Cap at 12 segments max. Document in pipeline.py.

**Production Failure Scenario:**
Gemini File API has upload limits. A 15-segment video triggers 15 parallel uploads. If 3 fail, the pipeline needs per-segment retry without blocking others. Current code has no retry isolation.

### Section 2: Code Quality Review

**Findings: 5 issues (1 P1, 2 P2, 2 P3)**

**[P1] (confidence: 9/10) schemas.py Missing `patterns` Field**
Design doc output schema has `patterns` with structure, pacing, visual_style, music_suggestion, reusable_template. But `ReplicationGuide.viral_patterns` is just `dict` with no structure. This means the Claude reasoner has no contract for what to fill in.

Decision: Add a `ViralPatterns` Pydantic model matching the design doc's patterns schema.

**[P2] (confidence: 8/10) gemini_analyzer.py `import` Inside Function**
Lines 26, 66, 102, 142: `import time` and `import json` are inside function bodies. Not a bug, but messy. These should be at module top level.

Decision: Move to top-level imports. Trivial fix.

**[P2] (confidence: 7/10) No Input Validation in video_processor.py**
`get_video_metadata` assumes video_stream exists (line 28: `video_stream["width"]`). If a non-video file is uploaded, this crashes with a confusing error.

Decision: Add explicit None check for video_stream. Raise ValueError with clear message.

**[P3] (confidence: 5/10) subprocess.run Without Timeout**
`video_processor.py:15,82,105`: `subprocess.run` calls to ffmpeg/ffprobe have no timeout. A corrupted video could hang forever.

Decision: Add `timeout=60` to all subprocess.run calls. Medium confidence this is an issue in practice.

**[P3] (confidence: 5/10) config.py Empty String Defaults for API Keys**
Lines 7-9: API keys default to empty strings. If `.env` is missing, the app starts but crashes on first API call with an unhelpful error.

Decision: Add startup validation that checks for empty API keys and fails fast with a clear message.

### Section 4: Performance Review

**Findings: 3 issues (1 P1, 1 P2, 1 P3)**

**[P1] (confidence: 9/10) Parallel Segment Analysis Not Implemented**
Design doc explicitly says "分段 Gemini 调用并行化是关键". Current code is synchronous. N segments = N * 10s sequential. A 10-segment video = 100s just for Gemini, blowing past the 90s target.

Decision: Use `asyncio.gather()` for N segment analyses in `pipeline.py`. This is the single most impactful change for hitting latency targets.

**[P2] (confidence: 7/10) ffmpeg Re-encoding Each Segment**
`video_processor.py:77`: `-c:v libx264 -preset fast` re-encodes every segment. For a 60s video with 6 segments, that's 6 full re-encodes. With `-c copy` (stream copy), segment splitting is near-instant.

Decision: Use `-c copy` for segment splitting. Only re-encode if the source has weird codec. Saves 3-5s total.

**[P3] (confidence: 5/10) No Disk Cleanup for Segment Files**
A 500MB video splits into 6 segments = ~500MB + 500MB original. After analysis, all files sit on disk. For a demo where multiple videos are analyzed, disk fills up fast.

Decision: Add cleanup step at end of pipeline. Delete segment files and extracted audio after analysis completes. Keep original video for potential re-analysis.

### Worktree Parallelization Strategy

| Step | Modules touched | Depends on |
|------|----------------|------------|
| 1. Pipeline orchestrator | backend/services/pipeline.py | schemas, video_processor, gemini_analyzer |
| 2. Transcriber service | backend/services/transcriber.py | config, schemas |
| 3. Claude reasoner service | backend/services/claude_reasoner.py | config, schemas |
| 4. FastAPI app + routes | backend/main.py, backend/api/ | pipeline, schemas |
| 5. Tests | tests/ | all services |
| 6. Frontend | frontend/ (Next.js) | backend API |

**Parallel lanes:**
- Lane A: Step 2 + Step 3 (independent services, no shared modules)
- Lane B: Step 1 (depends on schemas only, can start in parallel with A)
- After A+B merge: Step 4 (depends on pipeline)
- After Step 4: Step 5 (tests) + Step 6 (frontend) in parallel

**Execution order:** Launch A + B in parallel worktrees. Merge both. Then Step 4. Then Step 5 + 6 in parallel.
