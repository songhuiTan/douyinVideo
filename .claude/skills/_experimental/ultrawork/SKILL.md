# Ultrawork — 并行编排技能

> 借鉴 oh-my-openagent 的并行执行架构，为 MyHarness 添加多任务并行编排能力

---

## 触发条件

- 用户使用 `/ultrawork` 命令
- CLAUDE.md 识别到需求包含"并行"/"同时"/"ultrawork"/"批量"等信号
- Sprint PLAN 阶段识别到多个独立子任务

---

## 前置检查

1. 确认当前无活跃的阻塞任务（current-sprint.md 不是 CRITICAL 状态）
2. 确认并行任务数未超过限制（默认最多 5 个并行 Agent）
3. 确认每个子任务有明确的输入和输出定义

---

## 执行流程（5 Phase）

### Phase 1: DECOMPOSE（任务分解）

**目标**：将用户需求分解为可并行执行的独立子任务

```
输入：用户需求
  │
  ├─→ 识别所有独立工作单元
  ├─→ 分析依赖关系（哪些必须串行，哪些可以并行）
  ├─→ 为每个子任务分配类别：
  │     ┌───────────────────────────────────────────────────┐
  │     │  类别              │ 适用场景                      │
  │     │  research         │ 代码库探索、文档查阅           │
  │     │  implementation   │ 代码编写、功能开发             │
  │     │  review           │ 代码审查、质量检查             │
  │     │  testing          │ 测试编写、QA 验证              │
  │     │  documentation    │ 文档更新、注释补充             │
  │     └───────────────────────────────────────────────────┘
  ├─→ 构建任务依赖图（DAG）
  │
  └─→ 输出：Wave 划分
        Wave 1: [task_a, task_b, task_c]  ← 无依赖，立即启动
        Wave 2: [task_d]                  ← 依赖 Wave 1 结果
        Wave 3: [task_e]                  ← 依赖 Wave 2 结果
```

**依赖分析规则**：
- 修改不同文件 → 可并行
- 修改同一文件不同函数 → 可并行（但有合并风险，需标注）
- 一个任务的输出是另一个的输入 → 必须串行
- 共享依赖（如都读同一个 API schema）→ 可并行，但需共享上下文

### Phase 2: DISPATCH（并行分发）

**目标**：将 Wave 中的任务分发到并行 Agent 执行

```
对 Wave N 中的每个任务：
  │
  ├─→ 创建 Agent 调用（使用 Claude Code Agent 工具）
  │     {
  │       subagent_type: "general-purpose" | "Explore" | "Plan",
  │       description: "任务简述",
  │       prompt: "完整的任务上下文和指令",
  │       run_in_background: true（同一 Wave 内的任务并行执行）
  │     }
  │
  ├─→ 为每个 Agent 分配：
  │     ├── 明确的输入（文件路径、API 规格、设计图）
  │     ├── 明确的输出（期望的文件、格式、验收标准）
  │     ├── 完成标准（怎么算做完了）
  │     └── 超时限制（默认 10 分钟）
  │
  └─→ 记录到 docs/progress/parallel-tasks.md
```

**并发控制**：
- 默认最多 5 个并行 Agent
- 每个 Agent 使用 isolation: "worktree"（如果涉及代码修改）
- 研究类任务（Explore agent）不计入并发上限
- 同一文件的修改不能并行

### Phase 3: GATHER（结果汇聚）

**目标**：收集所有并行 Agent 的执行结果

```
等待 Wave N 的所有任务完成：
  │
  ├─→ 检查每个 Agent 的返回状态
  │     ├── completed → 提取输出
  │     ├── error → 记录错误，判断是否需要重试
  │     └── timeout → 取消任务，标记为超时
  │
  ├─→ 质量验证：
  │     ├── 输出是否满足验收标准
  │     ├── 是否引入了新的问题
  │     └── 是否与并行任务产生了冲突
  │
  └─→ 汇聚结果，准备传递给下一 Wave
```

**错误处理**：
- 单个 Agent 失败不阻塞其他 Agent
- 关键任务失败 → 整个 Wave 标记为部分失败
- 非关键任务失败 → 记录并继续
- 超时任务 → 取消并用更精确的 prompt 重试一次

### Phase 4: INTEGRATE（整合验证）

**目标**：将并行结果整合为完整交付物

```
所有 Wave 完成后：
  │
  ├─→ 合并并行产出物
  │     ├── 代码变更：检查冲突并合并
  │     ├── 研究结果：汇总关键发现
  │     └── 审查意见：按优先级排列
  │
  ├─→ 整体验证：
  │     ├── 运行测试套件（确保并行修改没有互相破坏）
  │     ├── 检查 import/依赖完整性
  │     └── 验证功能完整性（对照原始需求）
  │
  └─→ 输出整合报告
```

### Phase 5: COMPOUND（知识沉淀）

**目标**：从并行执行中提炼可复用模式

```
  ├─→ 记录并行执行效率（计划时间 vs 实际时间）
  ├─→ 识别并行模式：
  │     ├── 哪些类型的任务适合并行
  │     ├── 哪些依赖关系是意料之外的
  │     └── 并行导致了哪些冲突
  │
  ├─→ 更新 docs/progress/parallel-tasks.md 统计
  │
  └─→ 如果发现新的可复用并行模式 → 触发 /skill-evolution CAPTURED
```

## 熔断器机制

在并行执行过程中，监控每个 Agent 的行为：

| 信号 | 动作 |
|------|------|
| Agent 连续 5 次调用相同工具无进展 | 取消任务，记录原因 |
| Agent 总工具调用超过 100 次 | 取消任务，视为失控 |
| Agent 运行超过 10 分钟无输出 | 取消任务，标记超时 |
| 同一任务重试 2 次仍失败 | 放弃并行，回退到串行执行 |

**熔断后恢复**：
1. 分析失败原因（prompt 不够精确？任务过大？依赖缺失？）
2. 分解为更小的子任务
3. 用更详细的 prompt 重新分发
4. 如果仍然失败 → 回退到串行执行并通知用户

## 任务路由规则

根据任务类别选择最优的 Agent 类型：

| 任务类别 | Agent 类型 | 说明 |
|---------|-----------|------|
| research | Explore (quick) | 快速搜索，轻量级 |
| deep-research | Explore (very thorough) | 深度探索，全面分析 |
| implementation | general-purpose (worktree) | 代码编写，隔离环境 |
| review | general-purpose (readonly) | 代码审查，只读 |
| testing | general-purpose (worktree) | 测试编写，隔离环境 |
| planning | Plan | 架构规划，只读分析 |
| documentation | general-purpose | 文档编写 |

## 完成标志

- [ ] 所有 Wave 的任务已完成或已处理失败
- [ ] 整合验证通过（测试通过、无冲突）
- [ ] parallel-tasks.md 已更新
- [ ] 用户已收到整合报告
- [ ] 如有新发现，已触发 skill-evolution

## 错误恢复策略

| 错误 | 策略 |
|------|------|
| 并行 Agent 启动失败 | 重试 1 次，失败则串行执行 |
| Agent 返回不完整结果 | 用更精确的 prompt 补充执行 |
| 合并冲突 | 暂停并行，手动或用新 Agent 解决冲突 |
| 整体验证失败 | 逐个检查并行修改，定位引入问题的 Agent |
| 全部 Agent 超时 | 回退到串行模式，简化任务 |
